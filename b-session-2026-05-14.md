## B1 — CSRF mechanism cracked

In v6.15 the configuration-properties write attempts returned 403 "CSRF token missing or invalid". v6.15 documented this as a permission issue. **B1 corrects the diagnosis:**

```javascript
// WRONG (v6.15 inference):  the 403 means SYSTEM_ADMIN required
// CORRECT (v6.16 capture):  the 403 means missing CSRF header

const csrf = localStorage.getItem('CSRF');  // NOT from XSRF-TOKEN cookie
fetch('/rest/configuration-properties', {
  method: 'PUT',
  headers: { 'Content-Type': 'application/json', 'X-CSRF-TOKEN': csrf },  // exact header name
  body: JSON.stringify({...}),
});
```

With the correct `X-CSRF-TOKEN` header, `/rest/configuration-properties` returns **405 Method Not Allowed** — confirming the endpoint is GET-only on this URL. The actual write endpoint is presumably on `/MasterListsPage/commonproperties` or similar — to be captured by clicking the React admin page's Save button with a pre-load interceptor.

**v6.16 methodology rule:** Always interpret 403 with body `"CSRF token missing or invalid"` as a CSRF gate, not a permission gate. Use `localStorage['CSRF']` → `X-CSRF-TOKEN` header for all writes.

## B4 — Chain A Step 5 + Chain K (FHIR) — **Four FHIR bugs**

Probing FHIR endpoints surfaced four severe issues:

### FHIR-1: `/fhir/metadata` returns 500 — upstream proxy double-slash
```
GET /api/OpenELIS-Global/rest/fhir/metadata → 500
{"error":"Failed to query FHIR resources: HAPI-1359: Failed to parse response from server when performing GET to URL https://fhir.openelis.org:8443/fhir//metadata? - ca.uhn.fhir.parser.DataFormatException..."}
```
The proxy URL has a **double slash** (`/fhir//metadata`). Upstream FHIR base is `https://fhir.openelis.org:8443/fhir/` (trailing slash), proxy appends `/metadata`, producing `/fhir//metadata`. Fixable by stripping one slash.

### FHIR-2: `/fhir/Patient/27` returns 404 — patients not synced
```
GET /api/OpenELIS-Global/rest/fhir/Patient/27 → 404
{"error":"Failed to get FHIR resource: HTTP 404 : HAPI-1996: Resource Patient/27 is not known"}
```
Patient `Mana Pi` (patientPK=27) exists in the LIMS with persisted orders, but the upstream FHIR server doesn't have a corresponding Patient resource. Either patients aren't synced from LIMS → FHIR, or the FHIR ID isn't the LIMS patientPK.

### FHIR-3: `/fhir/Observation` returns 200 but with 151KB of broken HAPI internals
```
GET /api/OpenELIS-Global/rest/fhir/Observation → 200, 151,149 bytes
{
  "formatCommentsPre": [], "formatCommentsPost": [],
  "id": "Bundle/...",
  "meta": {
    "formatCommentsPre": [], "formatCommentsPost": [], ...
    "lastUpdatedElement": {
      "formatCommentsPre": [], "valueAsCalendar": 1778780755636,
      "nanos": 636000000, "secondsMilli": 55.636, "timeZoneZulu": false,
      "idElement": {
        "idElement": {
          "idElement": { ...recursive... }
        }
      }
    }
  }
  ...ends with ~200 closing braces
}
```
The Jackson serializer is dumping the **internal HAPI FHIR Java domain model** (including private fields like `formatCommentsPre`, `valueAsCalendar`, `disallowExtensions`, and recursive `idElement.idElement.idElement` due to a back-pointer) instead of invoking the FHIR JSON parser. Output is unusable for any real FHIR client.

### FHIR-4: `application/fhir+json` Accept header rejected with 406
```
GET /fhir/Observation, Accept: application/fhir+json → 406 HttpMediaTypeNotAcceptableException
GET /fhir/Observation, Accept: application/json      → 200 (with the FHIR-3 broken body)
```
The FHIR R4 spec requires servers to support `application/fhir+json`. Returning 406 on it is a compliance violation. Every standard FHIR client (which always sends `application/fhir+json` by default) will see all OpenELIS FHIR endpoints as unsupported.

### Module maturity correction
- **Previous rating (v6.14):** FHIR M3 on mgdev, "confirmed CapabilityStatement"
- **B-session evidence:** FHIR is M0–M1 on mgdev
- **Methodology lesson:** A passing CapabilityStatement read is INSUFFICIENT to rate FHIR M3. Full M3 requires Observation + Patient + Bundle round-trip with proper `application/fhir+json` content negotiation. v6.16 SKILL must update §FHIR rating criteria.

Full evidence: `outputs/b-session-fhir-evidence-2026-05-14.json`.

## Methodology updates for v6.16

1. **CSRF rule:** writes use `X-CSRF-TOKEN: localStorage['CSRF']`. 403 with `"CSRF token missing or invalid"` body means CSRF, not permission. Code in `_common.ts` should provide an `authedFetch()` helper.
2. **Interceptor placement:** to capture POST bodies in OpenELIS React app, install before the SPA's Axios module initializes. Monkey-patching `window.fetch` after page load misses captures because Axios binds its fetch ref at module init. Use a service worker, or pre-load fetch override via Playwright `addInitScript`.
3. **Results > By Order Accept footgun:** never click the inline "Accept" checkbox on Results > By Order. Validate via dedicated Validation module instead. Chain A spec and Persona PB/PC walk-throughs must reflect this.
4. **Same-URL GET/POST convention:** LogbookResults and AccessionValidation both serve GET (list) and POST (submit) on the same URL with method differentiation. The chain specs should document this pattern.
5. **FHIR rating criteria:** never rate FHIR M3 from CapabilityStatement alone. Require Observation + Patient + Bundle round-trip with `application/fhir+json`.
6. **Status enum transitions:** capture `analysisStatusId` enum values per state — observed 15 (Ready For Validation) → 6 (Validated).
