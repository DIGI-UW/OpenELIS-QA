## Failures Requiring Attention

### TC-09 — Add Sample / Place Order — FAILED

**Step that failed:** Step 4 — Submit order via POST to `/api/OpenELIS-Global/rest/SamplePatientEntry`

**Expected:** HTTP 200 with accession number issued; order appears in In Progress worklist

**Actual:** HTTP 500 response body: `"Check server logs"` — server-side exception, no accession number issued

**Severity:** High — blocks the entire order entry and result entry workflow

**Debugging performed (all attempts returned 500):**

The following combinations were all attempted across multiple sessions, all returning `500 "Check server logs"`:

| Attempt | Change Made | Result |
|---------|------------|--------|
| 1 | sampleXML had empty date/time → filled via UI calendar | 500 |
| 2 | `nextVisitDate` auto-populated to today → cleared via React state dispatch | 500 |
| 3 | `patientUpdateStatus: "UPDATE"` → changed to `"NO_ACTION"` | 500 |
| 4 | `currentDate: null` → set to `"23/03/2026"` via dispatch | 500 |
| 5 | `tests='446,445,443,444'` (QA tests) → `tests='4'` (Creatinine) | 500 |
| 6 | `sampleID='2'` → `sampleID='1'` | 500 |
| 7 | Switched to pre-existing patient (musa/muranga, patientPK=4) with COVID-19 IgG test (ID 305) | 500 |
| 8 | sampleXML time `'08:00'` → `'8:00'` (no leading zero) | 500 |

**Confirmed working:**
- CSRF token (`X-CSRF-Token`) is present and valid in all requests
- Patient record (patientPK=4) is valid — used in existing 2023 orders
- Test ID 305 (COVID-19 ANTIBODY IgG) is valid — appears in existing 2023 orders for this patient
- Site ID 3817 (AAAAAA - AADEMO0001) is valid
- Provider ID 939 (BOODOO, Mohammad Suhayl Qadr) is valid
- Payment option 1120 (Normal cash payment) and location 1303 (B1) are valid

**Root Cause — Confirmed Environment Bug:** After extended debugging, the server admin (Casey) also reproduced the error manually, using completely standard pre-existing tests (Western blot HIV, DENGUE PCR, HIV VIRAL LOAD, Creatinine) with a valid site and requester. The UI displayed: *"Oops, Server error please contact administrator."* This conclusively rules out any form-filling issue, QA test configuration, or accession number format as the cause. The `SamplePatientEntry` POST endpoint is broken at the server level for all users, with any combination of tests, on this environment. All 1820 existing orders in the system date from 2023. This is an infrastructure-level failure requiring server log investigation.

**Recommended Action:** Check server logs on `testing.openelis-global.org` for the Java stack trace emitted when POST `/api/OpenELIS-Global/rest/SamplePatientEntry` is called. The error likely reveals a NullPointerException, a database constraint violation, or a missing configuration entry.

### TC-REFERRAL — Referral Test Name Dropdown — PARTIAL (UI Bug)

**Step that failed:** Referral row — "Select Test Name" Carbon dropdown

**Expected:** Clicking or typing in the "Select Test Name" dropdown within the referral row should allow selection of a test to be referred; the field should either auto-populate from the ordered test or present a selectable list.

**Actual:** The Carbon `<Select>` component's `onChange` handler throws `e.onChange is not a function` on every interaction. Three approaches were attempted:
- Native `HTMLSelectElement.prototype` setter + change event: React overrides value back to empty
- Keyboard navigation (ArrowDown, letter keys): Carbon intercepts without updating selectedIndex
- Direct React fiber state dispatch with shallow spread: State dispatched but `defaultSelect` prop change does not re-render the selected option in the Carbon controlled Select

The form accepted the Submit with the referral test name field empty — no client-side validation blocks submission. The referral institute (CPHL) and sent date fields work correctly. The test name dropdown is the sole non-functional element in the referral row.

**Severity:** Medium — referral institute and date are captured; the test name is missing from the referral record

**Recommended Action:** Investigate the Carbon `Select` component usage in the referral row — the `onChange` prop may be receiving an incorrect context. Consider pre-populating `testName` from the ordered sample's test list, or replacing with a Carbon `Dropdown` (ComboBox) component which handles `onChange` correctly.

**Jira Ticket:** To be created — recommend filing as a UI bug against the referral row "Select Test Name" component
